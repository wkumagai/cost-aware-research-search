# Experiment Log

## 概要

このドキュメントは、Cost-Aware Research Search の初回実験ログです。
2つのランを実行し、「アイデア生成 → 実験実行 → GPT判定 → フィードバック改善」ループの動作を検証しました。

---

| 項目 | 値 |

| Iter | Score | テーマ | Key Finding |
|------|-------|--------|-------------|
| 1 | 6/10 | プロンプトフォーマットがLLM応答一貫性に与える影響 | JSON serialization エラーで実験未完了。修復もなし |

### 観察

- **Iter 1 → 2**: Iter 1 がエラーで失敗し、Iter 2 では簡略化されたが、テーマが変わりすぎた

- **Iter 3 → 4**: 有望だったCoTテーマを捨てて制約キーワードに逸脱。改悪

**問題点**: フィードバックが「改善」ではなく「別テーマへの逸脱」を引き起こしている

## 失敗分析

### 失敗の分類

| 失敗カテゴリ | 発生回数 | 説明 |
|-------------|---------|------|
| code_bug (JSON serialization) | 2 | `bool` や `numpy` 型のJSON変換失敗 |
| sample_too_small | 4+ | 毎回サンプルが15-40と少なすぎる |
| theme_fixation | 1 run | 同じテーマから抜け出せない |
| feedback_divergence | 1 run | フィードバックが改善でなく逸脱を引き起こす |
| stage0_bug | 全iter | `passed` フィールドが常にFalse (実装バグ) |

### 根本原因

1. **フィードバック→改善のマッピングが弱い**: 「サンプルを増やせ」と言われても、次のコード生成時にその指示が効かない。Claudeはidea specを改善するが、code genでは別の判断をする
2. **テーマ選択が自動でない**: 最初のテーマに固着するか、逸脱するかの二択。Thompson SamplingやMCTSがない状態では方向制御ができない
3. **Stage 0が機能していない**: バグで常にFAILだが、ループは続行してしまう
4. **修復が単純すぎる**: エラーメッセージをそのまま渡すだけでは修復率が低い

---

## 改善計画

### 即座に修正すべきもの

1. **Stage 0バグ修正** ✅ (修正済み、次ランから有効)
2. **コード生成プロンプトにフィードバックを直接注入**: idea specだけでなく、前回の判定結果もcode genに渡す
3. **サンプルサイズの最低値を強制**: idea specに `min_samples: 100` を追加

### 短期改善

4. **Thompson Sampling でテーマ選択**: 複数方向をarmとして持ち、固着を防ぐ
5. **修復ポリシーの強化**: 失敗パターン別の修復テンプレートを用意
6. **探索の多様性制約**: 連続2回同じテーマを選んだら、強制的に別方向へ

### 中長期

7. **MCTS による階層的探索**: テーマ → 仮説 → 実験設計 のツリー探索
8. **失敗メモリの活用**: 過去の失敗パターンをコード生成に注入
9. **Multi-fidelity の実装**: Stage 0-1 を厳格にして Stage 2 に進む候補を絞る

## 結論

初回実験から得られた最大の教訓:

1. **ジャッジの質は重要だが、それだけでは改善ループは回らない**
   GPT-5.4-proは的確なフィードバックを出すが、そのフィードバックがコード生成に伝搬しない

2. **テーマ固着は実際に起きる問題**
   RL的な探索（Thompson Sampling等）なしでは、同じ方向に固着するか、無関係な方向に逸脱する

3. **Multi-fidelity evaluationは有効になりうる**
   Stage 0 が正しく機能していればサンプルサイズ不足を事前に弾けたはず

4. **失敗は再利用可能なデータ**
   2ランで計8イテレーションの失敗パターンが蓄積された。これは次回の修復・回避に使える

次のステップとして、Stage 0修正 + フィードバック注入強化 + Thompson Sampling の追加を行う。
