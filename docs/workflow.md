# ワークフロー: 研究アイデア探索ループ

## このシステムは何をしているのか

このシステムは、**研究アイデアを自動で考え、実験し、評価し、改善する**、というサイクルを繰り返します。

人間の研究者が普段やっていることを、LLMを使って高速に回します:

1. アイデアを考える
2. 実験コードを書く
3. 実験を実行して結果を見る
4. 結果を振り返って、次のアイデアを改善する

---

## 各ステップの詳細

### ① アイデアを構造化する

人間が「こんな実験したい」と思うとき、頭の中にはぼんやりとしたイメージがあります。
このシステムでは、それを **typed idea spec** という決まった形式に変換します。

```yaml
hypothesis: "CoTプロンプトはembeddingのクラスタリング品質を改善する"
intervention_family: embedding_study
target_component: "sentence-transformersのembedding出力"
implementation_scope:
  lines_estimate: 150
  dependencies: [openai, sentence-transformers, scikit-learn]
  compute: cpu
  time_estimate_min: 5
proxy_evaluation:
  metric: silhouette_score
  baseline: direct_prompting
  success_threshold: ">0.05 improvement"
expected_failure_modes:
  - サンプルサイズ不足で有意差が出ない
  - API呼び出しエラー
rollback_plan: "サンプルを減らして最小実行を確認"
```

こうすることで:
- 何を検証するかが曖昧にならない
- 自動的にチェックできる
- 失敗しても何が悪かったかわかる

### ② 実行可能か事前チェック（Stage 0）

コードを書く前に、アイデア自体が実行可能かを確認します:

- YAML として正しいか？
- 仮説が書かれているか？
- 評価指標が定義されているか？
- CPU で動くか？
- 制限時間内に終わるか？

ここで弾かれたら、アイデア自体を再生成します。

### ③ 実験コードを自動生成

Claude が idea spec を読んで、完全に実行可能な Python スクリプトを生成します。

生成されるコードには:
- 必要な import
- データの準備（APIからの取得やローカル生成）
- 実験のメインロジック
- 結果の集計と統計分析
- JSON ファイルへの保存
- 結果テーブルの表示

### ④ 実験を実際に実行

生成されたコードを `subprocess` で実行します。
本物の API を呼び、本物のデータを処理し、本物の結果を出します。

**失敗した場合**: エラーメッセージを Claude に渡して、修復版のコードを生成してもらいます。
それでも失敗したら、「簡略化せよ」というフィードバック付きで次のイテレーションに進みます。

### ⑤ 結果を判定する

GPT-5.4-pro が研究者として結果を評価します。

**評価軸** (各1-10点):

| 軸 | 何を見るか |
|----|-----------|
| Novelty | 当たり前でない発見があるか |
| Rigor | 統計的に適切か、サンプルは十分か |
| Significance | 結果が重要か |
| Completeness | 明らかにやるべき追加実験をやったか |

さらに:
- **Key Finding**: 結果の1文まとめ
- **Improvement Suggestions**: 次に何を改善すべきか、具体的に3つ

### ⑥ フィードバックをもとにアイデアを改善

ここが「ループ」の核心です。

Claude に渡すのは:
- 前回の idea spec（全文）
- 今回のスコア（例: 3/10）
- Key Finding（例: 「サンプルが少なすぎて有意差が出なかった」）
- 改善提案（例: 「サンプルを100以上にしろ」「別のベースラインを追加しろ」）

Claude はこれを受け取って、**前回のアイデアを改善した新しい idea spec** を生成します。
ゼロからやり直すのではなく、前回の知見を活かして改善する点がポイントです。

## 具体例: 実際に何が起きたか

### Run 2 (GPT-5.4-pro判定) の4イテレーション

**Iter 1**: 「CoTプロンプトはembeddingクラスタリングを改善するか？」
- 15サンプルで実験 → 微小な差のみ
- 判定: 3/10 「サンプルが少なすぎる」

**Iter 2**: 前回のフィードバックを受けて改善を試みる
- silhouetteスコアで比較 → 0.081 vs 0.124
- 判定: 3/10 「まだサンプルが少ない、閾値未達」

**Iter 3**: さらに改善を試みる
- コードにバグ (JSON serialization error)
- 修復試行も失敗
- 判定: 2/10 「実験が完了していない」

**Iter 4**: 方向転換してドメイン特化クラスタリングへ
- medical/legal/generalの3ドメインで比較
- 結果が一貫しない (medical↓, legal↑)
- 判定: 3/10 「サンプル不足、一貫性なし」

### 何がうまくいかなかったか

1. **フィードバックがコードに伝わらない**: 「サンプルを増やせ」と言われるが、次のコード生成で15サンプルのまま
2. **同じ方向に固着**: 4回ともCoT/embedding関連で、別の有望な方向を試さない
3. **コードのバグ修復が弱い**: エラーメッセージを渡すだけでは修復できない場合がある

## 今後の改善方向

### すぐできること
- フィードバックをコード生成にも直接渡す
- サンプルサイズの最低値を強制する
- 同じテーマが3回続いたら強制的に別方向へ

### 次のフェーズ
- **Thompson Sampling**: 複数の研究方向を持ち、成績の良い方向を多く選ぶ
- **失敗メモリ**: 過去の失敗パターンを記録し、同じ失敗を繰り返さない
- **段階評価の厳格化**: Stage 0-1 を通過した候補だけを Stage 2 に進める

### 最終的な目標
- **MCTS**: アイデアをツリーとして展開し、有望な枝を深掘り
- **Offline RL**: 蓄積されたログから、良いアイデア変異のパターンを学習
